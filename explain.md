# Market Stress Tool 수정 내용 설명

이 문서는 통계와 코딩을 막 배우는 사람도 이번 수정 내용을 이해할 수 있도록 정리한 설명서입니다.

핵심은 단순합니다. 이 프로그램은 여러 경제·금융 지표를 모아서 “지금 시장이 평소보다 얼마나 불안한가?”를 계산합니다. 이번 수정은 그 계산을 더 현실적인 통계 방식으로 바꾸고, 사용자가 원자료와 과거 시점의 계산 결과를 더 쉽게 확인할 수 있게 만든 작업입니다. 이후 Streamlit에서 바로 쓰기 좋도록 기본 날짜, 탭 순서, 주요 사건 라벨 표시도 정리했습니다.

---

## 1. 이 프로그램이 하는 일

프로그램은 다음 흐름으로 작동합니다.

1. FRED, ECOS, Yahoo Finance 등에서 데이터를 가져옵니다.
2. 서로 날짜가 다른 데이터를 같은 날짜 축으로 맞춥니다.
3. 각 지표가 평소보다 얼마나 특이한지 z-score로 바꿉니다.
4. 비슷한 성격의 지표를 5개 채널로 묶습니다.
5. 5개 채널을 합쳐 종합 스트레스 점수를 만듭니다.
6. 현재 상태가 금리 충격, 위험 프리미엄 충격, 실물 침체, 공급 충격 중 어디에 가까운지 분류합니다.
7. 과거에 비슷했던 날짜와 이후 KOSPI 움직임을 보여줍니다.
8. 메인 시계열에는 미중 무역전쟁, 코로나, Fed 전환 같은 주요 사건 라벨을 날짜 위치에 맞춰 표시합니다.

여기서 중요한 점은 이 도구가 “주가를 맞히는 프로그램”은 아니라는 것입니다. 현재 시장 상태를 과거와 비교해 해석하는 보조 도구입니다.

---

## 2. 통계처리 개선

### 2.1 기존 z-score의 한계

보통 z-score는 아래처럼 계산합니다.

```text
z = (현재값 - 평균) / 표준편차
```

예를 들어 z-score가 `+2`라면 “평균보다 표준편차 2개만큼 높다”는 뜻입니다.

문제는 금융시장 데이터에는 극단값이 자주 나온다는 점입니다. 코로나, 금융위기, 전쟁, 정책 충격 같은 날에는 값이 매우 크게 튀기도 합니다. 이런 극단값이 평균과 표준편차를 흔들면, 이후의 정상적인 값도 이상하게 해석될 수 있습니다.

### 2.2 robust z-score로 변경

그래서 이번에는 기본 표준화 방식을 평균/표준편차가 아니라 `median/MAD` 방식으로 바꿨습니다.

- 평균 대신 중앙값을 사용합니다.
- 표준편차 대신 MAD를 사용합니다.
- 중앙값은 극단값 하나에 덜 흔들립니다.
- MAD도 극단값에 덜 민감한 산포 지표입니다.

쉽게 말하면, “일시적으로 튄 값 하나 때문에 평소 기준선 전체가 흔들리지 않게” 만든 것입니다.

관련 코드:

- `src/preprocessing/standardize.py`
- `rolling_robust_zscore()`
- `standardize_panel()`

### 2.3 z-score 클리핑

z-score가 너무 크게 튀면 한 변수가 전체 종합 점수를 과도하게 지배할 수 있습니다. 그래서 위험 방향을 적용하기 전에 z-score를 `-6`에서 `+6` 사이로 제한했습니다.

```text
예: z = 12.4 → 6.0으로 제한
예: z = -9.2 → -6.0으로 제한
```

이것은 데이터를 없애는 것이 아니라, 극단값 하나가 전체 판단을 망치지 않도록 상한선을 두는 방식입니다.

설정 위치:

- `config/variables.yaml`
- `z_clip_abs: 6.0`

### 2.4 백분위 계산 방식 개선

기존에는 z-score를 단순히 아래처럼 바꿨습니다.

```text
백분위 = 50 + 20 * z
```

이 방식은 직관적이지만, 실제 데이터 분포가 정규분포에 가깝다는 가정을 은근히 깔고 있습니다.

이번에는 `rolling empirical percentile rank`를 기본값으로 바꿨습니다. 말은 어렵지만 뜻은 단순합니다.

```text
최근 5년 데이터 중 현재값이 어느 정도 위치인가?
```

예를 들어 백분위가 97점이면, 최근 5년 분포에서 현재 스트레스가 상위 3% 수준이라는 뜻입니다.

관련 코드:

- `src/preprocessing/standardize.py`
- `rolling_percentile_rank()`
- `src/pipeline.py`

## 3. 임계값 보정 개선

### 3.1 왜 임계값 보정이 필요한가

예전에는 `1.5σ`, `2.0σ` 같은 기준을 주로 사용했습니다. 하지만 이런 기준은 해석하기 쉽다는 장점은 있어도, 실제 위기와 얼마나 잘 맞는지 확인이 필요합니다.

그래서 이번에는 두 가지 방식으로 임계값을 검증하도록 했습니다.

### 3.2 이벤트 라벨 기반 검증

과거 주요 스트레스 시점들을 지정하고, 그 주변 날짜를 “위기 구간”으로 표시합니다.

예:

- 2020-03-19: 코로나
- 2022-09-30: 영국 미니예산/강달러
- 2024-08-05: 일본 캐리 청산

그 다음 여러 threshold 후보를 테스트합니다.

```text
threshold가 낮으면 경보는 많이 울리지만 오경보도 많아짐
threshold가 높으면 오경보는 줄지만 위기를 놓칠 수 있음
```

그래서 아래 지표를 계산합니다.

- precision: 경보가 울렸을 때 실제 위기였던 비율
- recall: 실제 위기 중 경보가 잡아낸 비율
- F1: precision과 recall의 균형 점수

관련 코드:

- `src/analysis/threshold_calibration.py`
- `calibrate_event_thresholds()`
- `build_threshold_calibration()`

### 3.3 변동성 regime별 임계값

시장은 항상 같은 상태가 아닙니다. 평온한 장과 변동성이 큰 장에서는 같은 z-score도 다르게 해석해야 할 수 있습니다.

그래서 KOSPI의 63영업일 실현변동성을 기준으로 시장을 나눴습니다.

- 저변동
- 중변동
- 고변동

그리고 각 regime 안에서 q80, q90, q95 같은 경험적 분위수를 계산합니다.

```text
q90 = 해당 regime 안에서 상위 10% 수준의 스트레스
q95 = 해당 regime 안에서 상위 5% 수준의 스트레스
```

이 방식은 “현재 시장 환경에서 어느 정도면 높은 스트레스인가?”를 더 현실적으로 보여줍니다.

## 4. 새로 추가하거나 바꾼 탭

현재 탭 순서는 다음과 같습니다.

```text
메인 진단 → 세부 내용 → 과거 데이터 조회 → 과거 세부 내용 → 설명 → 원자료 시계열 → 통계 검증
```

### 4.1 과거 세부 내용 탭

처음 만든 과거 세부 내용 탭은 기간별 데이터 패널을 보는 성격이 강했습니다. 하지만 의도는 “특정 과거 날짜를 골랐을 때, 현재 세부내용 탭처럼 그날의 변수별 상태를 보는 것”이었습니다.

그래서 다시 고쳤습니다.

지금 과거 세부 내용 탭에서는 날짜를 하나 고르면 다음을 볼 수 있습니다.

- 그날 기준 raw 값
- 그날 기준 z-score
- 그날 기준 백분위
- 변수별 최근 1년 시계열
- 채널별 변수 묶음

관련 코드:

- `app.py`
- `render_history_detail_tab()`
- `src/ui/details_tab.py`
- `render_details_tab()`

### 4.2 원자료 시계열 탭

새로 `원자료 시계열` 탭을 추가했습니다.

이 탭은 처리되지 않은 데이터를 보여줍니다.

여기서 “처리되지 않은”의 뜻은 다음과 같습니다.

- 영업일 기준으로 날짜를 맞추기 전
- forward-fill 하기 전
- 전년 대비 변화율 같은 변환을 하기 전
- z-score로 표준화하기 전

즉, 로더가 외부 데이터 소스에서 가져온 원래 값에 가장 가까운 패널입니다.

관련 코드:

- `src/pipeline.py`
- `raw_panel`
- `app.py`
- `render_raw_timeseries_tab()`

### 4.3 기본 진단일자 변경

처음에는 기본 진단일자가 코드 안에 고정되어 있었습니다.

이제는 Streamlit 페이지가 실행되는 시점의 한국 날짜를 기준으로 직전 영업일이 자동으로 기본 진단일자가 됩니다.

예:

```text
2026-05-26 화요일에 페이지를 열면 → 2026-05-25 월요일
2026-05-25 월요일에 페이지를 열면 → 2026-05-22 금요일
일요일에 페이지를 열면 → 직전 금요일
```

한국 공휴일처럼 데이터가 없는 날짜는 파이프라인에서 실제 관측 가능한 직전 날짜로 다시 보정합니다.

관련 코드:

- `src/ui/date_defaults.py`
- `previous_business_day()`
- `app.py`

### 4.4 메인 시계열 이벤트 라벨

메인 진단 탭의 `종합 스트레스 시계열`에는 주요 시장 사건 라벨을 추가했습니다.

예:

- 미중 무역전쟁
- 코로나
- Fed 매파 전환
- 영국 미니예산/강달러
- 일본 캐리 청산
- 트럼프 상호관세

이 라벨은 단순히 화면에 고정된 글자가 아닙니다. 각 사건의 날짜를 기준으로 시계열 그래프의 x축 위치에 붙습니다. 그래서 사용자가 시작일과 종료일을 바꾸면 라벨도 해당 날짜 위치에 맞춰 함께 이동합니다.

처음에는 모든 라벨이 같은 높이에 표시되어 긴 기간을 조회하면 서로 겹칠 수 있었습니다. 그래서 라벨을 최대 3단 높이로 나눠 배치하도록 바꿨습니다.

```text
1번째 사건 라벨: 위쪽 줄
2번째 사건 라벨: 중간 줄
3번째 사건 라벨: 아래쪽 줄
4번째 사건 라벨: 다시 위쪽 줄
```

관련 코드:

- `src/analysis/threshold_calibration.py`
- `DEFAULT_CALIBRATION_EVENTS`
- `src/ui/charts.py`
- `make_composite_timeseries()`
- `tests/test_charts.py`

### 4.5 KOSPI/KOSDAQ 데이터 출처 변경

처음에는 KOSPI/KOSDAQ 타겟 데이터를 KRX API에서 가져왔습니다. 하지만 KRX API는 인증, 호출 제약, 캐시 관리 부담이 큽니다.

그래서 KOSPI/KOSDAQ 수집 경로를 Yahoo Finance로 바꿨습니다.

- KOSPI: `^KS11`
- KOSDAQ: `^KQ11`

이제 과거 유사 시점 이후 KOSPI 수익률 계산과 변동성 regime 계산은 KRX API가 아니라 yfinance에서 받은 지수 시계열을 사용합니다.

관련 코드:

- `config/variables.yaml`
- `src/data_collection/dispatcher.py`
- `src/data_collection/yfinance_loader.py`

### 4.6 통계 검증 탭

이 탭에서는 임계값 보정 결과를 볼 수 있습니다.

주요 내용:

- 현재 보정 레벨
- 현재 composite z-score
- 현재 변동성 regime
- 이벤트 라벨 기준 threshold 성능표
- regime별 q80/q90/q95 임계값

이 탭은 “왜 지금 위험 또는 주의라고 보는가?”를 통계적으로 점검하기 위한 화면입니다.

### 4.7 유가 하락 처리 개선

브렌트 유가는 처음에는 “크게 오르거나 크게 내리면 모두 위험”이라는 방식으로 처리했습니다. 하지만 한국 주식시장 관점에서는 유가 하락이 보통 기업 비용과 물가 부담을 낮추는 호재 또는 중립 요인입니다.

그래서 유가는 `positive_tail` 방식으로 바꿨습니다.

```text
유가 상승 모멘텀 → 공급충격 위험으로 반영
유가 하락 모멘텀 → 0으로 처리, 악재로 보지 않음
```

이렇게 해야 유가가 크게 떨어진 날이 단순히 “변동성이 크다”는 이유만으로 종합 스트레스 점수를 끌어올리는 문제를 줄일 수 있습니다.

## 5. 코딩 관점에서 바뀐 구조

### 5.1 파이프라인 결과 객체 확장

`run_full_diagnosis()`는 `DiagnosisResult`라는 결과 객체를 반환합니다. 이번에 이 객체에 여러 필드를 추가했습니다.

주요 추가 필드:

- `raw_panel`: 처리 전 원자료
- `stress_table`: 채널/종합 z-score 시계열
- `variable_pct_panel`: 변수별 백분위 시계열
- `calibration_summary`: 임계값 보정 요약
- `calibration_event_metrics`: 이벤트 라벨 기준 성능표
- `calibration_regime_thresholds`: regime별 분위수표

초보자 관점에서는 `DiagnosisResult`를 “계산 결과를 담는 상자”라고 보면 됩니다.

### 5.2 UI 헬퍼 추가

Streamlit 화면에서 반복해서 쓰는 작은 기능은 `src/ui/` 아래로 분리했습니다.

예:

- `src/ui/date_defaults.py`: 한국 날짜 기준 오늘과 직전 영업일 계산
- `src/ui/charts.py`: Plotly 차트 생성, 이벤트 라벨 표시, 라벨 높낮이 배치
- `src/ui/details_tab.py`: 현재/과거 세부 내용 탭에서 함께 쓰는 변수 상세 화면

이렇게 나누면 `app.py`가 너무 커지는 것을 줄이고, 날짜 계산이나 차트 배치 같은 기능을 따로 테스트할 수 있습니다.

### 5.3 화면과 계산 로직 분리

중요한 계산은 `src/analysis/`, `src/preprocessing/`, `src/pipeline.py`에 넣고, 화면은 `app.py`와 `src/ui/` 쪽에서 처리합니다.

이렇게 나누는 이유는 다음과 같습니다.

- 계산 로직을 테스트하기 쉬움
- 화면을 바꿔도 통계 계산이 망가지지 않음
- 나중에 Streamlit이 아닌 다른 UI로 바꾸기도 쉬움

### 5.4 테스트 추가

새 통계 보정 로직은 테스트도 추가했습니다.

관련 파일:

- `tests/test_threshold_calibration.py`
- `tests/test_pipeline.py`

## 6. 이번 수정의 의미

이번 수정의 의미는 크게 세 가지입니다.

첫째, 통계처리가 금융시장 데이터의 현실에 더 가까워졌습니다. 평균/표준편차만 쓰는 방식보다 fat-tail과 극단값에 덜 흔들립니다.

둘째, 프로그램이 더 투명해졌습니다. 사용자는 최종 점수만 보는 것이 아니라 원자료, 변환 후 값, 과거 시점의 세부값까지 확인할 수 있습니다.

셋째, 임계값을 “그냥 정한 숫자”로 두지 않고, 과거 이벤트와 시장 regime 기준으로 검증할 수 있게 됐습니다.

넷째, 실제 사용성이 좋아졌습니다. 페이지를 열 때마다 전 영업일이 기본 진단일자로 잡히고, 메인 시계열에서 주요 사건을 같이 볼 수 있어 점수가 왜 움직였는지 직관적으로 확인할 수 있습니다.

---

## 7. 아직 남은 한계

이번 개선으로 프로그램이 더 좋아졌지만, 완벽해진 것은 아닙니다.

남은 한계는 다음과 같습니다.

- 과거 위기 이벤트 수가 아직 적습니다.
- 이벤트 라벨은 사람이 정한 것이므로 주관성이 있습니다.
- 표본외 검증이 더 필요합니다.
- 한국 특화 변수는 아직 부족합니다.
- k-NN 유사 시점은 인과관계가 아니라 유사도 비교입니다.

따라서 이 도구는 “확정적 예측기”가 아니라 “시장 스트레스 상태를 구조적으로 설명해주는 보조 도구”로 보는 것이 맞습니다.

```bash
streamlit run app.py
```

테스트 실행:

```bash
pytest -q
```

```text
243 passed, 7 skipped, 13 warnings
```
